# v1.4.12 — BTC Address Hardening: Manual Test Guide

## What this version does

Two changes:

1. **Pre-publish address freshness check.** When you publish an invoice with a BTC address, the app calls `GET https://mempool.space/api/address/<addr>` (network-aware via `NEXT_PUBLIC_BTC_NETWORK`) and rejects the publish if the address has any on-chain or mempool activity. If mempool.space is unreachable, the check fails open: the publish proceeds and a warning is logged.

2. **Network-aware mempool URLs in emails.** The "View transaction on mempool.space" link in payment-detected and payment-confirmed emails now respects `NEXT_PUBLIC_BTC_NETWORK`. On testnet, the link points to `mempool.space/testnet4/tx/...` instead of the mainnet URL (which would 404 or, worse, show an unrelated mainnet tx with the same id).

What was *not* changed: invoices are still hard-deleted (no soft-delete). The freshness check covers the deletion-reuse risk because any paid-and-deleted address now has on-chain history and gets rejected on the next attempt to reuse it.

---

## Setup

```bash
set -a; source .env; set +a
```

You will need:
- A draft invoice you can publish.
- A known BTC address that has on-chain history (any address that has ever received bitcoin — for testnet, easiest is to grab one from a faucet or a previously-funded wallet).
- A known BTC address that has *no* history (a fresh address from your wallet).
- The dev server running. Confirm `NEXT_PUBLIC_BTC_NETWORK` in `.env` — most tests below assume `testnet4`.

---

## TEST 1 — Fresh address publishes successfully

**Steps:**
1. Create a draft invoice. Tick "Accept Bitcoin" and paste in a fresh address (no on-chain history on the configured network).
2. Click **Publish**.

**Expected:**
- Publish succeeds. Invoice transitions to `pending`.
- No error toast.
- Server logs do not contain `mempool.space unreachable`.

---

## TEST 2 — Address with prior on-chain history is rejected

**Steps:**
1. Create a draft invoice. Tick "Accept Bitcoin" and paste in an address you know has prior on-chain activity.
2. Click **Publish**.

**Expected:**
- Publish fails. Error message: **"This address has already received transactions — use a fresh address for each invoice."**
- Invoice stays in `draft`.

**Verifying the right address class:** open `https://mempool.space/<network>/address/<addr>` in a browser. If "Confirmed received" or "Mempool received" is non-zero, it should be rejected.

---

## TEST 3 — Address with pending mempool tx is rejected

**Steps:**
1. Find an address that has a pending unconfirmed transaction in the mempool (no confirmed history yet). On testnet, you can broadcast a small tx to a fresh address and immediately try to publish before it confirms.
2. Paste it into a draft invoice and publish.

**Expected:**
- Same rejection message as TEST 2.
- The freshness rule treats pending mempool txs the same as confirmed ones.

(If you can't easily produce a pending-only address, this case is also covered by the unit test in `src/lib/mempool.test.ts`.)

---

## TEST 4 — Mempool down → publish proceeds with a warning

**Steps:**
1. Block outbound connectivity to mempool.space, the easiest way being:
   ```bash
   # In a temporary shell
   sudo sh -c 'echo "127.0.0.1 mempool.space" >> /etc/hosts'
   ```
   (Remember to revert this after the test.)
2. Restart the dev server so any cached DNS clears.
3. Create a draft with any BTC address (fresh or otherwise — doesn't matter).
4. Click **Publish**.

**Expected:**
- Publish succeeds. Invoice transitions to `pending`.
- Server console contains a line like:
  ```
  [publish] mempool.space unreachable, address history check skipped for invoice <invoice-id>
  ```
- The uniqueness check (against other active invoices in the DB) still runs and is still authoritative.

**Cleanup:**
```bash
sudo sh -c "sed -i '' '/127.0.0.1 mempool.space/d' /etc/hosts"
```

## TEST 5 — Email mempool link respects testnet

**Steps:**
1. Set `NEXT_PUBLIC_BTC_NETWORK=testnet4` in `.env` and restart the dev server.
2. Publish an invoice with a fresh testnet address.
3. Trigger a payment to that address (testnet faucet or your own wallet) and let the payment-detected email fire.
4. Open the email.

**Expected:**
- The "View transaction on mempool.space" link points to `https://mempool.space/testnet4/tx/<txid>`.
- Clicking the link loads a real testnet transaction page (not a 404, not a mainnet tx).

## TEST 6 — Email mempool link is correct on mainnet

**Steps:**
1. Set `NEXT_PUBLIC_BTC_NETWORK=mainnet` (or remove the env var entirely) and restart.
2. Use a real mainnet payment flow (or stub one), and inspect the resulting email.

**Expected:**
- The "View transaction on mempool.space" link points to `https://mempool.space/tx/<txid>` (no `testnet4` segment).
- Loads a real mainnet transaction page.

## TEST 7 — UI and email URLs agree

**Steps:**
1. With a payment detected on testnet, compare:
   - The mempool link rendered on the public invoice page (the "View address on mempool.space" link in the mark-as-sent dialog or the tx link wherever it appears in the UI).
   - The mempool link in the payment-detected email.

**Expected:**
- Both URLs share the same host and `/testnet4` path segment for the same network.
- This is also locked down by automated parity tests in `src/lib/email/send.test.ts` ("renders a network-aware mempool URL ...").

## TEST 8 — Uniqueness check still works (regression)

**Steps:**
1. Publish invoice A with address X.
2. Create draft invoice B with address X.
3. Click **Publish** on B.

**Expected:**
- B fails to publish with: **"btc_address: This bitcoin address has already been used on invoice <ref>. Please provide a unique address."**
- This rule is independent of mempool.space — it should fire even if mempool is unreachable.

---

## TEST 9 — Save-draft rejects a poisoned address (v1.4.12 hotfix)

**Steps:**
1. Open the new-invoice page or an existing draft.
2. Tick "Accept Bitcoin" and paste in an address with prior on-chain history.
3. Click **Save** (not Publish).

**Expected:**
- Save fails immediately with: **"This address has already received transactions — use a fresh address for each invoice."**
- The draft is not created (or, if editing, the existing record is unchanged).
- This must fire on save, not just on publish.

---

## TEST 10 — Drafts do not start a payment watcher (v1.4.12 hotfix)

**Steps:**
1. Create a draft invoice with a BTC address (use a fresh address so save passes).
2. Open the dashboard detail page for that draft.
3. Open the browser dev-tools console and the network tab.

**Expected:**
- No WebSocket connection to mempool.space.
- No HTTP polling against `mempool.space/api/address/...`.
- No POSTs to `/api/invoices/<id>/payment-status`.
- The status shown is the static "Draft" badge (no live status component).

Compare against publishing the same invoice and reloading: now you should see the WebSocket connect and polling activity start.

## TEST 11 — payment-status route refuses to flip a draft (v1.4.12 hotfix)

This is the load-bearing fix. Reproduces the data-corruption scenario.

**Steps:**
1. Create a draft invoice (any status, any address).
2. From the terminal, manually POST to the route as the watcher would:
   ```bash
   curl -i -X POST http://localhost:3000/api/invoices/<draft-id>/payment-status \
     -H "Content-Type: application/json" \
     -d '{"txid": "deadbeef", "status": "paid"}'
   ```
3. Repeat with `<archived-id>` (mark another invoice as archived first via the UI).

**Expected:**
- Both calls return **HTTP 409** with body `{"error":"Invoice is not in a payable state","status":"draft"}` (or `"archived"`).
- The invoice's status in the DB is unchanged.
- No payment_confirmed / payment_detected email is sent.

For comparison, the same call against a `pending` invoice with a real matching tx should still return 200 and complete the transition.

## Rollback notes

If you need to roll this version back:
- No DB migrations were added in v1.4.12 — only application code changed.
- Reverting the merge commit fully restores prior behaviour.
- The `NEXT_PUBLIC_MEMPOOL_BASE_URL` env var that was previously read by `src/lib/email/send.ts` is no longer consulted; you can remove it from `.env` and Vercel.
