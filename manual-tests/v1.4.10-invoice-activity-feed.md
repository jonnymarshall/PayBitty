# v1.4.10 — Invoice Activity Feed: Manual Test Guide

## What this version does

The invoice detail page's **Email Activity** card has been generalised into a single **Activity** card. In addition to the existing email events (`invoice_published`, `payment_detected`, `payment_confirmed`), the card now also shows three manual state transitions:

- **Marked as sent** — `publishAndMarkSent` server action (Mark-as-sent, Download-and-mark-as-sent)
- **Marked as paid** — `markPaid` server action
- **Marked as overdue** — `markOverdue` server action

These manual events are written server-side to a new `invoice_events` table (migration `0013_invoice_activity_events.sql`) using the service role. The card fetches both `email_events` and `invoice_events` in parallel and merges them into one chronological feed (most-recent-first).

Each event type has its own icon:

| Event | Icon (lucide) |
|---|---|
| Email sent / queued / skipped | `Mail` |
| Email failed | `AlertCircle` (red) |
| Marked as sent | `Send` |
| Marked as paid | `CheckCircle` (green) |
| Marked as overdue | `Clock` |
| Marked as unpaid | `RotateCcw` |

Failures to insert an `invoice_events` row are logged but do **not** block the primary state transition — the same defensive pattern used by the email send pipeline.

## One-time setup

```bash
set -a; source .env; set +a
sb_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"; }
```

Restart the dev server so the migration takes effect and the renamed component bundle ships.

You will need:
- A working `RESEND_API_KEY` and a verified `EMAIL_FROM` for tests that involve sending real emails (TEST 1).
- Existing draft invoices you can publish / mark as sent / mark as paid / mark as overdue.

## TEST 1 — Email event renders in the new Activity card

**Proves:** existing `email_events` rows still render correctly under the new heading and icon scheme.

**Setup:**
1. Create a draft invoice with a non-empty `client_email` you control.
2. From the detail page, click **Send now via email** (or do this from `/invoices/new`'s split-button menu).
3. Wait for the green "Email queued for delivery" banner.

**Run:** Stay on `/invoices/$INV`.

**Expect:**
- The card's heading reads **Activity** (NOT "Email Activity").
- One row inside the card with:
  - A **`Mail`** envelope icon on the left.
  - Label `Invoice published — <recipient>`.
  - Relative timestamp on the right (e.g. `Apr 29, 3:14 PM`).
  - Hovering the timestamp shows the absolute ISO timestamp in a tooltip.
- No per-row inner borders, no expandable rows — tight glanceable feed.

**DB confirmation:**
```bash
sb_select "email_events?invoice_id=eq.$INV&select=email_type,status,created_at" | jq
```
Returns a `sent` row.

**DB confirmation:**
```bash
sb_select "invoice_events?invoice_id=eq.$INV_S&select=event_type,created_at" | jq
```
Returns `[{ "event_type": "marked_as_sent", ... }]`.

## TEST 5 — All five icons coexist, ordered most-recent-first

**Proves:** the merge logic combines email and manual events into one chronological list, and each event type has the right icon.

**Setup:** Create a draft invoice. Copy its id: `INV_ALL=<id>`.

**Run (in this exact order, with a few seconds between actions so timestamps separate cleanly):**
1. From the form, **Send now via email** (real email send).
2. Reload the detail page. Click **Mark as paid**.
3. Reload. Click **Mark as overdue** (yes, even on a paid invoice — we're just exercising the action; you can revert via SQL after).
4. Reload `/invoices/$INV_ALL`.

**Expect:** The Activity card now shows (top → bottom):
1. **`Clock`** icon · `Marked as overdue` · *most recent*
2. **green `CheckCircle`** icon · `Marked as paid`
3. **`Mail`** icon · `Invoice published — <recipient>` · *oldest*

(If a `payment_detected` event also appeared because the invoice happened to receive BTC, that's fine — the test is about ordering and icon distinctness, not exact row count.)

## TEST 7 — Insert failure does not block the primary state transition (smoke test)

**Proves:** if the `invoice_events` insert fails for any reason, `markPaid` (and the other actions) still flip the invoice status. This is the safety net described in the spec ("Failures to insert an `invoice_events` row are logged but do not block the primary state transition").

**Setup:** This requires temporarily breaking the insert. The cheapest way:

```bash
# Temporarily revoke insert privileges from the service role on invoice_events.
# Run this in the Supabase SQL editor or via psql with admin creds:
revoke insert on invoice_events from service_role;
```

Pick a pending invoice. Copy its id: `INV_BREAK=<id>`.

**Run:**
1. Visit `/invoices/$INV_BREAK` and click **Mark as paid**.
2. Reload the page.

**Expect:**
- The status badge **does** flip to **Paid** (the primary transition succeeded).
- The Activity card does **not** show a `Marked as paid` row (the insert failed, gracefully).
- The dev-server log shows a `[invoice-events] failed to record marked_as_paid …` console error.

**Cleanup (very important — restore privileges):**
```sql
grant insert on invoice_events to service_role;
```

## TEST 8 — RLS on invoice_events: owner can read, anon cannot

**Proves:** the `owner can read own invoice events` policy works; the absence of an anon insert policy is enforced.

**Run:**
```bash
# Anon read (should be empty):
curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/invoice_events?select=id" \
  -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $NEXT_PUBLIC_SUPABASE_ANON_KEY" | jq

# Anon insert (should fail with an RLS / permission error):
curl -s -X POST "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/invoice_events" \
  -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $NEXT_PUBLIC_SUPABASE_ANON_KEY" \
  -H "Content-Type: application/json" \
  -d '{"invoice_id":"00000000-0000-0000-0000-000000000000","user_id":"00000000-0000-0000-0000-000000000000","event_type":"marked_as_paid"}' | jq
```

**Expect:**
- First call: `[]`.
- Second call: an error about a missing policy / row-level security violation. If this succeeds, RLS is broken — file a bug.

## TEST 9 — No backfill: pre-v1.4.10 invoices show only their email events

**Proves:** the migration deliberately did NOT backfill historical manual transitions; older invoices show only what was recorded going forward.

**Run:** Visit any invoice that was marked as paid / overdue **before this branch was deployed**.

**Expect:**
- The Activity card still renders (assuming there are email events).
- It shows **only** email-event rows. No `Marked as paid` row appears even though `invoices.status='paid'`.

This is by design — backfilling from `invoices.sent_at` etc. was explicitly out of scope.

## Cleanup

```bash
# Wipe invoice_events rows for the test invoices used in TESTS 2–5 (replace ids):
for INV in "$INV_S" "$INV_P" "$INV_O" "$INV_ALL"; do
  curl -s -X DELETE "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/invoice_events?invoice_id=eq.$INV" \
    -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" \
    -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"
done
```

> Deleting from `invoice_events` does not roll back the underlying `invoices` row — `status`, `sent_at`, etc. were set independently. If you need a fully clean slate for re-testing, also reset the relevant columns on the invoice row (e.g. `update invoices set status='pending', sent_at=null, send_method=null where id='…'`).
