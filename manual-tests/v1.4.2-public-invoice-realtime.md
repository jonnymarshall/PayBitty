# v1.4.2 — Public Payer Page Live Updates: Manual Test Guide

## What this version does

Until v1.4.1, the public `/invoice/[id]` page had no Supabase Realtime subscription. If the background cron (path C) — or any other server-side path — flipped an invoice's status while the payer was looking at the page, their badge wouldn't move until they refreshed. The on-page mempool WebSocket (path A) only sees transactions hitting the watched address; it can't see DB-level transitions made by anyone else.

v1.4.2 closes that gap. The public page now subscribes to Supabase Realtime UPDATE events for its specific invoice id, using the **anon** key (the payer is unauthenticated). When any other path (cron, fast-path API, owner action) writes to the row, the new `status` is pushed to the page within ~1 second and applied to local React state — no refresh needed.

The on-page mempool watcher remains the fastest path for transactions hitting the watched address. Realtime is the catch-all for everything else.

---

## One-time setup

### Step 1 — Verify the migration is applied

```bash
set -a; source .env; set +a
sb_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"; }
sb_update() { curl -s -X PATCH "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY" -H "Content-Type: application/json" -H "Prefer: return=representation" -d "$2"; }
sb_anon_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" -H "Authorization: Bearer $NEXT_PUBLIC_SUPABASE_ANON_KEY"; }
now_iso() { date -u +%Y-%m-%dT%H:%M:%SZ; }
run_cron() { curl -s -H "Authorization: Bearer $CRON_SECRET" "http://localhost:3000/api/cron/payment-sweep" && echo; }
```

> The `sb_anon_select` helper hits Supabase as an unauthenticated user — same key the browser uses on the public page.

### Step 2 — Confirm the policy exists

```bash
curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/invoices?select=id&limit=1" \
  -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $NEXT_PUBLIC_SUPABASE_ANON_KEY"
```

**Expect:** a JSON array (possibly empty). If you see `{"code":"42501","message":"permission denied"}` or `{"message":"new row violates row-level security policy"}`, the migration didn't apply — re-run `npx supabase db push`.

### Step 3 — Restart the dev server

The new hook lives in a client component, so Next.js needs a fresh build to ship the new bundle. `Ctrl-C` your dev server, then `npm run dev` again.

## The tests

### TEST 1 — Anon RLS allows SELECT only on non-draft invoices

**Proves:** the new policy from migration `0009` exposes only published invoices to anonymous clients. Drafts stay owner-only.

**Behind the scenes:** policy `anon_select_non_draft` on `invoices` is `for select to anon using (status != 'draft')`. The Realtime channel applies the same RLS check before delivering payloads.

**Run:**
1. Create a new invoice but **don't publish it** — leave it as a draft. Copy its ID: `DRAFT_ID=<id>`.
2. Create a second invoice and **publish it**. Copy its ID: `PUB_ID=<id>`.
3. In the terminal:
```bash
sb_anon_select "invoices?id=eq.$DRAFT_ID&select=id,status" | jq
sb_anon_select "invoices?id=eq.$PUB_ID&select=id,status" | jq
```

**Expect:**
- Draft query returns `[]` (RLS blocks the row).
- Published query returns `[{ "id": "...", "status": "pending" }]`.

> If the draft query returns the row, the policy is too permissive — file a bug.

---

### TEST 2 — Cron-driven transition moves the payer's badge without refresh

**This is the core fix.** Proves that a server-side status change reaches the open payer page in real time.

**Behind the scenes:** the public page subscribes to `postgres_changes` on `invoices` filtered to its own id. The cron updates the row → Postgres emits a logical replication event → Supabase Realtime broadcasts it → the page applies `payload.new.status` to local React state.

**Run:**
1. Create and publish a new testnet invoice. Copy its ID: `INV_ID=<id>` and the BTC address.
2. Open `/invoice/<id>` in a browser. Confirm the status badge says **Pending**. Leave the tab visible — do not refresh.
3. **Don't broadcast a real payment.** We're testing the DB → page push, not the mempool watcher. Force the cron to think a payment was detected by directly setting the status:
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"status\":\"payment_detected\",\"btc_txid\":\"deadbeef\",\"mempool_seen_at\":\"$(now_iso)\"}" > /dev/null
```
4. **Within ~1 second**, watch the badge.

**Expect:** the badge flips from **Pending** to **Payment Detected** without any refresh, navigation, or click. The QR/address area also auto-reveals (because `showPaymentDetails` is true for non-payable statuses).

5. Repeat with `paid`:
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"status\":\"paid\"}" > /dev/null
```

**Expect:** badge flips to **Paid** within ~1 second.

> If the badge doesn't move: open DevTools → Console. A `[public-invoice-realtime] CHANNEL_ERROR` line means the Realtime websocket couldn't authenticate — usually missing `NEXT_PUBLIC_SUPABASE_ANON_KEY` or the policy didn't apply. A `CHANNEL_ERROR: timeout` means the migration is missing.

---

### TEST 3 — End-to-end with the real cron

**Proves:** the production code path works hands-off, not just synthetic DB writes.

**Run:**
1. Reset the invoice from Test 2:
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"status\":\"pending\",\"stage_attempt\":0,\"mempool_seen_at\":null,\"next_check_at\":\"$(now_iso)\",\"btc_txid\":null}" > /dev/null
```
2. Open `/invoice/<id>` in the browser. Status: **Pending**.
3. From a testnet wallet, send a small amount to the invoice's BTC address. **Do not open any other tab.** The point is to prove the cron — not the on-page watcher — drives the transition.

   *(If you want to be extra strict, open the page in a second browser without internet access to mempool.space. The Realtime channel goes through Supabase, not mempool.)*
4. Wait ~30 seconds for mempool propagation, then trigger the cron from your terminal:
```bash
run_cron
```

**Expect:**
- `run_cron` returns `{"processed":1,"transitions":1,"errors":0}`.
- The badge on the open `/invoice/<id>` page flips to **Payment Detected** within ~1 second of the cron returning, without any refresh.

---

### TEST 4 — Owner-driven mark-as-paid pushes to the payer

**Proves:** any DB-level transition reaches the payer, not just cron-driven ones.

**Behind the scenes:** there is currently no "Mark as paid" action exposed in the dashboard UI in v1.4.x — but the same code path runs whenever the owner triggers anything that updates the row (e.g. archiving, future actions). For now, simulate it directly:

**Run:**
1. Reset and re-publish the test invoice as in Test 3, step 1.
2. Open `/invoice/<id>` in the payer browser. Status: **Pending**.
3. In another tab, log into the dashboard and visit the invoice on `/invoices/<id>`.
4. In the terminal, simulate an owner-driven status flip:
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"status\":\"payment_detected\",\"btc_txid\":\"deadbeef\"}" > /dev/null
```

**Expect:**
- The owner's `/invoices/<id>` page badge updates within ~1 second (path D, already worked).
- The payer's `/invoice/<id>` page badge **also** updates within ~1 second (path E, new in v1.4.2).

---

### TEST 5 — Mempool watcher (path A) still wins when present

**Proves:** Realtime is additive — it doesn't break or replace the existing fastest-path WebSocket watcher.

**Behind the scenes:** both `PaymentWatcher` and `usePublicInvoiceRealtime` call `setStatus`. Whichever fires first wins; the other is a no-op (the status is already updated). The mempool watcher typically beats Realtime by a few hundred ms when the tx is broadcast.

**Run:**
1. Reset and re-publish the test invoice. Open `/invoice/<id>`.
2. Open DevTools → Network → filter `WS` (WebSocket).
3. From a testnet wallet, send a small amount to the invoice's BTC address.

**Expect:**
- Two WebSockets visible: one to `mempool.space` (path A), one to your Supabase project's Realtime endpoint (path E).
- Within ~1–2 seconds of broadcast, the badge flips to **Payment Detected**. The `mempool.space` socket should be the one that fires the message that triggers the change (visible as an `address-transactions` frame in the WS message log).
- A POST to `/api/invoices/<id>/payment-status` returns `200`.

---

### TEST 6 — `visibilitychange` safety net refreshes a stale tab

**Proves:** if the Realtime websocket silently dropped while the tab was backgrounded, returning to the tab triggers `router.refresh()` so the server fetch pulls the latest row.

**Run:**
1. Open `/invoice/<id>` for any pending invoice. Status: **Pending**.
2. Switch to another tab or another window — make sure the invoice tab is **not** the visible foreground.
3. In the terminal, change the status:
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"status\":\"paid\"}" > /dev/null
```
4. Switch back to the invoice tab. (Don't refresh manually.)

**Expect:** the badge shows **Paid** within ~1 second of returning, even if the Realtime push was missed while backgrounded. (In practice the live socket usually delivers it too — this safety net is the belt-and-braces for socket drops, mobile background-throttling, and laptop sleep.)

---

### TEST 7 — Hook does not break a `notFound()` page

**Proves:** the hook gracefully handles the no-invoice case (the page server-renders a 404 before the hook ever runs, but worth a quick check).

**Run:** Visit `/invoice/00000000-0000-0000-0000-000000000000` in a browser.

**Expect:** standard Next.js 404 page. No console errors, no orphaned websockets in DevTools → Network → WS.

## Quick smoke test (3 minutes)

If you only have time for a minimal pass, run: **1, 2, 6**. Test 1 proves the RLS policy is correct, test 2 proves the realtime push works end-to-end, test 6 proves the visibility-change safety net catches dropped sockets. Together those cover the whole of v1.4.2's value.
