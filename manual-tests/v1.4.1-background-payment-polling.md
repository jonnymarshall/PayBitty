# v1.4.1 — Background Payment Polling: Manual Test Guide

## What this version does

Until v1.4, payment detection required someone to have a page open — either the payer at `/invoice/[id]` (client-side mempool watcher) or the invoice owner at `/invoices` (login sweep). If both closed their tabs, a payment could sit undetected until someone logged back in.

v1.4.1 replaces the login sweep with a **Vercel Cron** endpoint that polls mempool.space on a per-invoice schedule. Detection now works with nobody logged in. The tests below prove this end-to-end.

---

## One-time setup

### Step 1 — Fix your `.env`

Open `.env` and make sure `CRON_SECRET` has a **real value** (not `<anything>`):

```
CRON_SECRET=local-dev-cron-secret
```

*Any string works. The server just checks the request's `Authorization: Bearer …` matches this.*

### Step 2 — Restart the dev server

Next.js only reads `.env` at startup. In the terminal running `npm run dev`: `Ctrl-C`, then `npm run dev` again.

### Step 3 — Open a second terminal for the tests

Everything below runs in a separate terminal from the dev server. Paste this **once per session** — it loads `.env` into your shell and defines helpers that wrap Supabase's REST API:

```bash
set -a; source .env; set +a
sb_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"; }
sb_update() { curl -s -X PATCH "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY" -H "Content-Type: application/json" -H "Prefer: return=representation" -d "$2"; }
now_iso() { date -u +%Y-%m-%dT%H:%M:%SZ; }
run_cron() { curl -s -H "Authorization: Bearer $CRON_SECRET" "http://localhost:3000/api/cron/payment-sweep" && echo; }
```

> **Why `run_cron`, not `cron`?** `cron` is the name of macOS's system cron daemon, and your shell will run that instead of our function if we use the same name.

### Step 4 — Verify setup

```bash
echo "$CRON_SECRET"
```
Must print your secret (e.g. `local-dev-cron-secret`). If it's empty, Step 3 failed — re-paste.

```bash
type run_cron
```
Must print the function body (not "run_cron: not found"). If it's missing, re-paste Step 3.

```bash
run_cron
```
Must print: `{"processed":0,"transitions":0,"errors":0}` (or similar counts if invoices are already queued).

- If it prints `{"error":"Unauthorized"}`, your shell's `$CRON_SECRET` doesn't match the `.env` file. Re-run `set -a; source .env; set +a`, then try again.
- If it prints a **404 HTML page**:
  - First run `type run_cron` and confirm the URL in the function body is **not** followed by `\;` — some terminals escape the semicolon during paste, which gets baked into the URL. If you see `\;`, re-paste the single-line helper with the URL quoted.
  - If the function is clean but the dev server still 404s, nuke the turbopack cache: `Ctrl-C` the dev server, then in `/Users/jonny/code/paybitty` run `rm -rf .next && npm run dev`. Wait for `Ready in XXXms` and retry.

## The tests

### TEST 1 — Cron endpoint is bearer-protected

**Proves:** random attackers can't trigger the sweep. Only requests with the correct `Authorization` header get through.

**Behind the scenes:** the route checks `process.env.CRON_SECRET` against the incoming header. Vercel attaches this header automatically on scheduled runs. Wrong or missing → 401.

**Run:**
```bash
curl -i http://localhost:3000/api/cron/payment-sweep
curl -i -H "Authorization: Bearer wrong-value" http://localhost:3000/api/cron/payment-sweep
curl -i -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/payment-sweep
```

**Expect:** first two respond `HTTP/1.1 401 Unauthorized`. Third responds `HTTP/1.1 200 OK` with JSON `{"processed":…, "transitions":…, "errors":0}`.

---

### TEST 2 — `publishInvoice` initialises schedule columns

**Proves:** when a draft is published, it's immediately enrolled in the background polling rotation.

**Behind the scenes:** the `publishInvoice` server action now writes `status='pending'` **plus** `next_check_at = now()+1min`, `stage_attempt=0`, `mempool_seen_at=null`. That tells the cron "start checking this invoice in one minute."

**Run:**
1. In the UI, create a new draft invoice with a valid testnet BTC address. Publish it.
2. Copy the invoice ID from the URL (`/invoices/<id>`).
3. In the terminal:
```bash
INV_ID=<paste-id-here>
sb_select "invoices?id=eq.$INV_ID&select=status,stage_attempt,mempool_seen_at,next_check_at" | jq
```

**Expect:**
```json
[
  {
    "status": "pending",
    "stage_attempt": 0,
    "mempool_seen_at": null,
    "next_check_at": "2026-04-24T00:42:00.000Z"
  }
]
```
(The `next_check_at` timestamp should be roughly 1 minute in the future.)

### TEST 3 — Cron advances a pending invoice with no payment

**Proves:** the cron reaches its target invoice, runs the decision logic, updates the schedule, and backs off through the pre-mempool delays (1m → 5m → 10m → 30m → stop).

**Behind the scenes:** each cron tick calls `decidePaymentSchedule(invoice, txs, now)`. With no paying tx found, it increments `stage_attempt` and sets `next_check_at` to the next delay in the `PRE_MEMPOOL_DELAYS_MS` table. After the fourth attempt it returns `next_check_at=null` → polling stops.

**Run:**

Use the same `INV_ID` from Test 2 (make sure no testnet payment has been sent to it).

```bash
# Force the cron to fire immediately (skip the 1-min wait)
sb_update "invoices?id=eq.$INV_ID" "{\"next_check_at\":\"$(now_iso)\"}" > /dev/null
run_cron
sb_select "invoices?id=eq.$INV_ID&select=stage_attempt,next_check_at" | jq
```

**Expect after first run:** `stage_attempt=1`, `next_check_at ≈ now + 5 minutes`.

Repeat the same three-line block three more times. Expected progression:

| Run | `stage_attempt` | `next_check_at` |
|-----|-----------------|-----------------|
| 1   | 1               | +5 min          |
| 2   | 2               | +10 min         |
| 3   | 3               | +30 min         |
| 4   | 4               | `null` (stop)   |

After run 4, the row won't be picked up again — polling has gracefully exhausted.

### TEST 4 — Cron detects an unconfirmed payment (the core fix)

**Proves:** the payer can close the tab, the owner can stay logged out, and the system still detects the payment from mempool.

**Behind the scenes:** the cron fetches all txs for the invoice's BTC address from mempool.space. If one pays to the address, `decidePaymentSchedule` transitions `pending → payment_detected`, stamps `mempool_seen_at=now`, resets `stage_attempt=0`, and schedules the next check at `now+10min` (post-mempool cadence). Then the route dispatches a "Payment detected" email to the owner.

> **Why "trigger the cron" is a test step.** Vercel Cron does **not** fire against `localhost` — it only runs on deployed Vercel environments. In production, the schedule in `vercel.json` makes Vercel hit this endpoint every minute automatically, so the detection would happen hands-off within ~60 seconds of broadcast. In local dev, nothing is scheduled to hit localhost, so we play Vercel's role manually by calling `run_cron`. Alternatively, open a throwaway terminal and run `while true; do run_cron; sleep 60; done` to simulate the production schedule — then closing the tab + broadcasting + waiting ≤ 60 s gets the email hands-off.

**Run:**
1. Create and publish a **new** testnet invoice. Copy its ID: `INV_ID=<new-id>`.
2. From any testnet wallet, send a small amount to the invoice's BTC address. **Do not wait for confirmation** — just broadcast.
3. Close any `/invoice/[id]` browser tab you opened for it (to prove the client-side watcher is not involved).
4. Trigger the cron (stand-in for Vercel):
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"next_check_at\":\"$(now_iso)\"}" > /dev/null
run_cron
sb_select "invoices?id=eq.$INV_ID&select=status,mempool_seen_at,stage_attempt,next_check_at,btc_txid" | jq
```

**Expect:**
- `run_cron` returns `{"processed":1,"transitions":1,"errors":0}`.
- Row shows `status="payment_detected"`, `mempool_seen_at` ≈ now, `stage_attempt=0`, `next_check_at` ≈ now+10min, `btc_txid` populated.
- **A "Payment detected" email arrives** in the invoice owner's inbox. No "Payment confirmed" email yet.

### TEST 5 — Cron upgrades detected → paid on confirmation

**Proves:** once the tx confirms on-chain, the next cron tick promotes the invoice to `paid`, stops polling, and sends the final email.

**Behind the scenes:** `decidePaymentSchedule` sees a confirmed tx → transitions to `paid` with `next_check_at=null`. The cron dispatches a "Payment confirmed" email (and no duplicate "detected").

**Run:**

Using the same `INV_ID` from Test 4, wait for the tx to confirm. Track progress at:
```
https://mempool.space/testnet/tx/<btc_txid from test 4>
```

Once you see 1+ confirmations:
```bash
sb_update "invoices?id=eq.$INV_ID" "{\"next_check_at\":\"$(now_iso)\"}" > /dev/null
run_cron
sb_select "invoices?id=eq.$INV_ID&select=status,next_check_at" | jq
```

**Expect:**
- `run_cron` returns `{"processed":1,"transitions":1,"errors":0}`.
- Row shows `status="paid"`, `next_check_at=null`.
- **A "Payment confirmed" email arrives.** No duplicate "detected" email.

### TEST 6 — Fast path still works and uses the same scheduler

**Proves:** the client-side mempool watcher (`/invoice/[id]` page) still triggers real-time transitions, and it now updates the same schedule columns as the cron, so the two systems stay in sync.

**Behind the scenes:** the POST to `/api/invoices/[id]/payment-status` now delegates to `decidePaymentSchedule` (same helper the cron uses), so whichever path fires first writes the same row shape.

**Run:**
1. Create and publish a **new** testnet invoice. `INV_ID=<new-id>`.
2. Open `/invoice/<id>` in a browser with DevTools Network tab open.
3. From a testnet wallet, send a small amount to the invoice's BTC address.
4. In DevTools, watch for a POST to `/api/invoices/<id>/payment-status` returning 200.
5. In the terminal:
```bash
sb_select "invoices?id=eq.$INV_ID&select=status,mempool_seen_at,stage_attempt,next_check_at" | jq
```

**Expect after the 0-conf event:** `status="payment_detected"`, `mempool_seen_at` set, `stage_attempt=0`, `next_check_at` ≈ +10min (same shape as Test 4).

**Expect after the 1-conf event** (re-run the `sb_select` command): `status="paid"`, `next_check_at=null`.

### TEST 7 — Login sweep is gone

**Proves:** the v1.4 login sweep has been fully removed — no stale code left behind.

**Behind the scenes:** the dashboard layout used to render `<LoginSweepTrigger />`, which fired a server action on mount. Both the component and the server action were deleted.

**Run:**
1. In the app, log out and log back in via magic link.
2. Land on `/invoices`. Open DevTools → Network → filter `fetch/xhr`.
3. Scroll the request list. There should be **no** request mentioning `sweep`.
4. Confirm the codebase doesn't reference the old code:
```bash
grep -r "LoginSweepTrigger\|runLoginSweep\|sweep-action" src/ 2>/dev/null
```

**Expect:** empty output (no matches anywhere).

## Cleanup

To reset a test invoice to pending for further polling tests:

```bash
sb_update "invoices?id=eq.$INV_ID" "{\"status\":\"pending\",\"stage_attempt\":0,\"mempool_seen_at\":null,\"next_check_at\":\"$(now_iso)\",\"btc_txid\":null}" > /dev/null
```

To see every invoice currently enrolled in polling:
```bash
sb_select "invoices?next_check_at=not.is.null&select=id,status,stage_attempt,next_check_at&order=next_check_at.asc" | jq
```
