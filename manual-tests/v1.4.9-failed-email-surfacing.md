# v1.4.9 — Failed Email Surfacing: Manual Test Guide

## What this version does

When the most recent `invoice_published` email attempt for an invoice has `status='failed'`, the failure is surfaced on the `/invoices` list as a small red `AlertCircle` indicator next to the status badge (tooltip: *"Email failed to send to this client"*). The detail page already shows the failure reason inside the **Email Activity** card (introduced in v1.4.3), so v1.4.9 deliberately does not duplicate that signal at the top of the detail page.

Behind the scenes, the list page now reads from a new view `invoice_email_summary` (migration `0012`) that left-joins each invoice to its most-recent `invoice_published` row in `email_events`.

## One-time setup

```bash
set -a; source .env; set +a
sb_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"; }
```

Restart the dev server so the migration takes effect and the new component bundle ships.

You will need at least three invoices in the dashboard:
- **A — failed publish** (created in TEST 1)
- **B — successful publish** (any past invoice that was published with `RESEND_API_KEY` set and `EMAIL_FROM` verified — pick one whose Email Activity card shows a green **Sent** badge)
- **C — never attempted** (a draft, OR a non-draft published before v1.4.3 / via "Mark as sent" without firing email)

---

## TEST 1 — Per-row indicator appears on a failed-publish row

**Proves:** when a row's most recent `invoice_published` email_event has `status='failed'`, the dashboard list shows a red `AlertCircle` next to the status badge.

**Setup (creates invoice A):**
1. In `.env`, set `EMAIL_FROM` to an unverified domain, e.g. `Paybitty <noreply@notmydomain.example>`. Restart the dev server.
2. Create a new invoice with a non-empty `client_email` (any address you control). Click **Send now via email**. Copy its id: `INV_A=<id>`.
3. Revert `EMAIL_FROM` to your verified value and restart the dev server (so other tests aren't affected).

**Run:** Visit `/invoices`.

**Expect:**
- Invoice A's row shows the **Pending** badge **followed by a small red `!` (AlertCircle) icon** in the Status column.
- Hovering the icon shows the tooltip *"Email failed to send to this client"*.

**DB confirmation:**
```bash
sb_select "invoice_email_summary?id=eq.$INV_A&select=last_publish_email_status,last_publish_email_error" | jq
```
Returns `[{ "last_publish_email_status": "failed", "last_publish_email_error": "Resend validation_error: ..." }]`.

## TEST 2 — No indicator on rows whose last email succeeded

**Proves:** the indicator is conditional on `status='failed'`.

**Run:** On `/invoices`, locate invoice B (the one with a successful past send).

**Expect:**
- Invoice B's row shows the status badge **followed by the standard `Mail` (envelope) icon** — the at-a-glance "sent via email" indicator from v1.4.8.
- **No** red `AlertCircle` icon.

---

## TEST 3 — No indicator on rows that never attempted email

**Proves:** rows with no `invoice_published` row in `email_events` show no indicator at all.

**Run:** On `/invoices`, locate invoice C.

**Expect:**
- If invoice C is a draft: no trailing icon at all (drafts have no `send_method`).
- If invoice C was marked as sent manually: shows the **`HandHelping`** icon next to the status badge — not the failed indicator.
- **No** red `AlertCircle` icon in either case.

---

## TEST 4 — Failed indicator replaces (does not stack with) the sent-method icon

**Proves:** on a row that was *sent via email* but the send failed, only the failed indicator shows — not both icons stacked together.

**Setup:** Invoice A from TEST 1 already meets this criterion: it was attempted via "Send now via email" (so `send_method='email'`) and Resend rejected it (so `last_publish_email_status='failed'`).

**Run:** On `/invoices`, look at invoice A's Status column.

**Expect:**
- Exactly **one** trailing icon: the red `AlertCircle`.
- **Not** the `Mail` (envelope) icon and the `AlertCircle` side by side.

---

## TEST 5 — Detail page does NOT add a duplicate alert

**Proves:** v1.4.9 deliberately does not surface the failure at the top of the detail page; the existing **Email Activity** card is the only place the failure reason appears on `/invoices/[id]`.

**Run:** Visit `/invoices/$INV_A`.

**Expect:**
- **No** banner / alert / red box at the top of the page next to the status badge.
- Scroll down to the **Email Activity** card. The failed attempt is shown there as a row with the red **Failed** badge and the error reason in small red text below — exactly as it has been since v1.4.3.

---

## TEST 6 — The view respects RLS

**Proves:** `invoice_email_summary` inherits the underlying `invoices` and `email_events` RLS policies — owners only ever see their own data.

**Run:**
```bash
curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/invoice_email_summary?select=id" \
  -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $NEXT_PUBLIC_SUPABASE_ANON_KEY" | jq
```

**Expect:** `[]`. The anon key has no logged-in user, so RLS on `invoices` returns zero rows and the view returns nothing. If you see rows here, RLS is broken — file a bug.

## Quick smoke test (60 seconds)

If you only have time for the minimum: **TEST 1** + **TEST 4**. Test 1 proves end-to-end that a failed Resend send surfaces correctly on the list (DB row, view, indicator). Test 4 proves the single-icon rule.

## Cleanup

```bash
# Optional: delete the failed-publish row so invoice A goes back to "no attempt" state
curl -s -X DELETE "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/email_events?invoice_id=eq.$INV_A&email_type=eq.invoice_published" \
  -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" \
  -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"
```

> Note: deleting the `email_events` row will not re-enable the "Send via email" button — that's gated by `invoices.email_attempted_at`, which is set at attempt-time and is intentionally not affected by this view. To fully reset for re-testing, you'd also need to clear `email_attempted_at` on the invoice row.
