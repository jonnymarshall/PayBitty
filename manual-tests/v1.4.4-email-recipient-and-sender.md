# v1.4.4 — Email Recipient Expansion + Sender Identity: Manual Test Guide

## What this version does

The **payment-detected** and **payment-confirmed** emails now go to **both** the invoice owner and the payer (`client_email`) — two separate Resend calls per status transition, with role-specific templates. The owner gets "Your client paid invoice X"; the payer gets "Your payment to {sender} has been detected / confirmed". If `client_email` is blank on an invoice, only the owner receives email — the payer-side send is silently skipped.

Separately, `EMAIL_FROM` and the Supabase custom SMTP **Sender** are now both `SatSend <team@mail.satsend.me>`, so all transactional and auth mail arrives with one consistent `From:` identity.

---

## One-time setup

```bash
set -a; source .env; set +a
sb_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"; }
sb_update() { curl -s -X PATCH "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY" -H "Content-Type: application/json" -H "Prefer: return=representation" -d "$2"; }
run_cron() { curl -s -H "Authorization: Bearer $CRON_SECRET" "http://localhost:3000/api/cron/payment-sweep" && echo; }
```

You'll want **two real inboxes** you can read — one as the owner (the account you're signed in with) and one as the payer (`client_email`). The two-recipient behaviour is the whole point of this release; testing with `client_email` blank only covers the skip path.

Restart the dev server so the new templates and route columns ship.

**Proves:** transactional mail leaves with the new `team@mail.satsend.me` identity.

**Run:**

```bash
grep -E '^EMAIL_FROM=' .env
```

**Expect:** exactly `EMAIL_FROM="SatSend <team@mail.satsend.me>"`. If it differs, fix it and restart the dev server before continuing.

**Also verify in the Supabase dashboard:** Project Settings → Auth → SMTP Settings → **Sender** shows `team@mail.satsend.me`. (No code change here — this is a dashboard setting.) Triggering a magic-link login from the app and checking the `From:` of the resulting email is the easiest way to confirm.

## TEST 2 — `payment_detected` fires two emails (one per role)

**Proves:** a single `pending → payment_detected` transition results in two distinct Resend calls — owner-framed and payer-framed.

**Run:**
1. Sign in as the **owner** account. Publish a testnet invoice with `client_email` set to your **payer** inbox. Copy `INV_ID=<id>`.
2. Send a small amount (e.g. 1000 sats) of testnet BTC to the invoice's `btc_address` — any testnet faucet works (e.g. https://coinfaucet.eu/en/btc-testnet/). Wait ~10s for the tx to hit mempool. Then trigger the cron:
   ```bash
   run_cron
   ```

   *Alternative (fast-path):* if you already know the txid (e.g. from your wallet or mempool.space), POST it directly:
   ```bash
   curl -s -X POST "http://localhost:3000/api/invoices/$INV_ID/payment-status" \
     -H "Content-Type: application/json" \
     -d '{"txid":"<real-testnet-txid>","status":"payment_detected"}'
   ```
   The route validates the tx against mempool.space and rejects (`"Transaction does not pay to invoice address"`) any txid that isn't a real tx paying to this invoice's address — there's no fake-it path.

**Expect:**
- **Owner inbox** receives an email with subject `Your client paid invoice <INV-NUMBER>`. Body opens "{client_name} just sent {total} for invoice X…" and links to **Open in Paybitty** (dashboard view).
- **Payer inbox** receives an email with subject `Your payment for invoice <INV-NUMBER> has been detected`. Body opens "Your payment of {total} to {senderName} for invoice X has been broadcast…" and links to **View invoice** (the public `/invoice/[id]` page, no login needed).
- DB confirms both rows:
  ```bash
  sb_select "email_events?invoice_id=eq.$INV_ID&email_type=eq.payment_detected&select=recipient,status" | jq
  ```
  Returns two rows, one per recipient, both `status: "sent"`.

## TEST 3 — `payment_confirmed` also fires two emails

**Proves:** the same dual-recipient behaviour applies to the confirmation transition.

> ⚠️ **Don't `sb_update` the status to `paid` directly.** `sb_update` writes to the DB and bypasses the route handler — and the email dispatch lives **in the route handler**, not in a DB trigger. There's no recovery either: once the row is `paid`, the fast-path's status-order guard short-circuits any future POST and the cron only scans `pending` / `payment_detected`, so the invoice never gets another chance to email. Always drive the transition through the API.

**Run:** continuing from TEST 2's invoice (which is now `payment_detected` with `btc_txid` set), drive the `paid` transition through the route, using the txid already stored on the row:
```bash
TXID=$(sb_select "invoices?id=eq.$INV_ID&select=btc_txid" | jq -r '.[0].btc_txid')

curl -s -X POST "http://localhost:3000/api/invoices/$INV_ID/payment-status" \
  -H "Content-Type: application/json" \
  -d "{\"txid\":\"$TXID\",\"status\":\"paid\"}"
```

(Or wait for the testnet tx to confirm and let the cron escalate it via `run_cron`. Both paths run the same email-dispatch code.)

**Expect:**
- **Owner inbox** subject: `Invoice <INV-NUMBER> confirmed on-chain`.
- **Payer inbox** subject: `Your payment for invoice <INV-NUMBER> is confirmed`.
- DB:
  ```bash
  sb_select "email_events?invoice_id=eq.$INV_ID&email_type=eq.payment_confirmed&select=recipient,status" | jq
  ```
  Returns two `sent` rows.

## TEST 4 — Blank `client_email` skips the payer send (owner still receives)

**Proves:** the payer-side dispatch is silently skipped when `client_email` is empty, and the owner copy still goes out.

**Run:**
1. Publish a fresh testnet invoice with **`client_email` left blank**. Copy `INV_ID`.
2. Send testnet sats to the invoice's `btc_address` (faucet → ~10s → `run_cron`), or POST a real testnet txid via the fast-path. See TEST 2 for both methods.

**Expect:**
- **Owner inbox** receives the "Your client paid invoice X" email.
- No payer email exists (because there's no payer address).
- DB confirms exactly one row was written for this transition:
  ```bash
  sb_select "email_events?invoice_id=eq.$INV_ID&email_type=eq.payment_detected&select=recipient" | jq
  ```
  Returns a single row whose `recipient` is the owner's account email.

## TEST 5 — Each path writes its own pair (no duplicates) [optional]

**Proves:** the DB-level `eq('status', priorStatus)` guard still prevents both the fast-path and the cron from sending the same pair twice when they race.

> ⚠️ This test needs a **real testnet txid** that already pays to the invoice's `btc_address`. The route rejects any made-up txid against mempool.space, so the race never reaches the email-dispatch code. The race assertion is also covered by the unit test for the route's status guard — skip this test unless you specifically want to see the dedup happen with real on-chain data.

**Run:**
1. Publish a testnet invoice with `client_email` set. Copy `INV_ID`.
2. Send testnet sats to the invoice's `btc_address` (faucet works) and grab the resulting `TXID` from mempool.space once it appears in mempool.
3. Race two fast-path calls with that real txid (simulating a duplicate watcher / cron race):
   ```bash
   TXID=<the-real-testnet-txid>
   for i in 1 2; do
     curl -s -X POST "http://localhost:3000/api/invoices/$INV_ID/payment-status" \
       -H "Content-Type: application/json" \
       -d "{\"txid\":\"$TXID\",\"status\":\"payment_detected\"}" &
   done; wait
   ```
4. `run_cron` once.

**Expect:** exactly **two** `payment_detected` rows in `email_events` for this invoice (one owner, one payer) — not four. The losing call short-circuits on `PGRST116` before reaching email dispatch.
```bash
sb_select "email_events?invoice_id=eq.$INV_ID&email_type=eq.payment_detected&select=recipient" | jq 'length'
```
