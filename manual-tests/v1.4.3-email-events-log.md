# v1.4.3 — Email Event Log: Manual Test Guide

## What this version does

Every transactional email the app tries to send (`invoice_published`, `payment_detected`, `payment_confirmed`) now writes a row to the new `email_events` table — `queued` on insert, then flipped to `sent` / `failed` / `skipped_no_api_key` once Resend responds. Owners can see the log on `/invoices/[id]` in a new **Email Activity** card.

## One-time setup

```bash
set -a; source .env; set +a
sb_select() { curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY"; }
sb_update() { curl -s -X PATCH "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/$1" -H "apikey: $SUPABASE_SERVICE_ROLE_KEY" -H "Authorization: Bearer $SUPABASE_SERVICE_ROLE_KEY" -H "Content-Type: application/json" -d "$2"; }
now_iso() { date -u +%Y-%m-%dT%H:%M:%SZ; }
```

Restart the dev server so the new component bundle ships.

## TEST 1 — Publish an invoice, see a `sent` row

**Proves:** the happy path — `safeSend` writes a `queued` row, Resend succeeds, the row flips to `sent` with a `resend_message_id`.

**Run:**
1. With `RESEND_API_KEY` set in `.env`, create and publish an invoice with a **non-empty `client_email`** you can receive at. (If `client_email` is blank, `publishInvoice` skips the send entirely and no row is written — that's correct behaviour, but it'll leave the activity card empty and look like a bug.) Copy its id: `INV_ID=<id>`.
2. Visit `/invoices/$INV_ID` in the dashboard.

**Expect:**
- An **Email Activity** card appears below the actions, listing one row: **Invoice published** → recipient → green **Sent** badge → timestamp.
- The DB confirms it:
```bash
sb_select "email_events?invoice_id=eq.$INV_ID&select=email_type,status,resend_message_id" | jq
```
Returns `[{ "email_type": "invoice_published", "status": "sent", "resend_message_id": "..." }]`.

## TEST 2 — Drafts show no card

**Proves:** the card is hidden when there are no events (i.e., drafts).

**Run:** Create a draft invoice (don't publish). Visit its `/invoices/[id]` page.

**Expect:** No **Email Activity** section anywhere on the page.

---

## TEST 3 — Failed send shows a red badge with hover error

**Proves:** Resend errors are captured as `status='failed'` with `error_message` populated, and the UI surfaces the reason on hover.

**Run:**
1. Set `EMAIL_FROM` in `.env` to an unverified domain (e.g. `Paybitty <noreply@notmydomain.example>`). Restart the dev server.
2. Publish a new invoice **with a non-empty `client_email`** — without one, `publishInvoice` short-circuits before calling `safeSend` and no row is written. Any address you can receive at is fine; the test is about Resend rejecting our `from`, not about delivery. Copy `INV_ID`.
3. Visit `/invoices/$INV_ID`.

**Expect:**
- The activity row shows **Invoice published** → red **Failed** badge.
- Below the row, in small red text, the Resend error message is visible (e.g. *"Resend validation_error: The notmydomain.example domain is not verified..."*).
- DB:
```bash
sb_select "email_events?invoice_id=eq.$INV_ID&select=status,error_message" | jq
```
Returns `status: "failed"` with `error_message` populated.

> Don't forget to revert `EMAIL_FROM` after this test.

## TEST 4 — Missing `RESEND_API_KEY` records `skipped_no_api_key`

**Proves:** local dev without Resend still works and is auditable.

**Run:**
1. Comment out `RESEND_API_KEY` in `.env`. Restart the dev server.
2. Publish a new invoice **with a non-empty `client_email`** (same reason as TEST 3). Copy `INV_ID`.

**Expect:**
- Activity card row: **Invoice published** → yellow **Skipped** badge.
- No actual email is sent.
- DB row: `status = 'skipped_no_api_key'`, `resend_message_id = null`, `error_message = null`.

> Restore `RESEND_API_KEY` after this test.

## TEST 5 — Payment transitions add their own rows

**Proves:** the cron and fast-path callsites also pass `user_id` and write `email_events` rows.

**Run:**
1. With `RESEND_API_KEY` set, publish a testnet invoice. Copy `INV_ID`.
2. Send testnet sats to the invoice's BTC address (or simulate via `sb_update` setting `status='payment_detected'`). Wait for the cron tick (or trigger it manually).
3. Once on-chain (1+ confirmations), wait for a second cron tick — or `sb_update` to `status='paid'`.
4. Visit `/invoices/$INV_ID`.

**Expect:** the activity card lists, newest first:
- **Payment confirmed** → green **Sent** → `your-account@…`
- **Payment detected** → green **Sent** → `your-account@…`
- **Invoice published** → green **Sent** → `client@…`

(If you used `sb_update` to skip Bitcoin, the payment emails go to the owner's account email — confirm those land in the inbox you logged in with.)

## TEST 6 — RLS isolation: another owner can't see your events

**Proves:** the policy `owner can read own email events` works.

**Behind the scenes:** `email_events` has RLS enabled with a single SELECT policy `auth.uid() = user_id`. There's no anon SELECT policy, so unauthenticated requests should also see nothing.

**Run:**

```bash
curl -s "$NEXT_PUBLIC_SUPABASE_URL/rest/v1/email_events?select=id" \
  -H "apikey: $NEXT_PUBLIC_SUPABASE_ANON_KEY" \
  -H "Authorization: Bearer $NEXT_PUBLIC_SUPABASE_ANON_KEY" | jq
```

**Expect:** `[]`. If you see rows here, RLS is broken — file a bug. (The service-role helpers in this doc bypass RLS by design; the anon key is what matters for isolation.)

**Cross-owner UI check (optional):**
1. Sign in as user A, publish an invoice. Confirm the **Email Activity** card on `/invoices/<A's id>` shows rows.
2. Sign out. Sign in (or sign up) as user B.
3. Visit `/invoices/<A's id>` directly in the URL bar.

**Expect:** Next.js renders 404 — the `invoices` RLS policy blocks the row before the page can read it, so user B can't even land on the invoice, let alone its events.
