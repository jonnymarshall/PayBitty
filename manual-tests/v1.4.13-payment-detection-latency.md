# v1.4.13.x — Payment Detection Latency: Manual Test Guide

> **Visual markers used in this doc:**
> - 🆕 = test added in v1.4.13.2 or later
> - ✏️ = test changed/expectations updated in v1.4.13.2 or later
> - 🗑️ = test removed (mechanism it tested no longer exists)
> - (no marker) = unchanged from initial v1.4.13 release

## What this version does

The "paid but didn't click the button" scenario was taking 60–120 seconds end-to-end. Four patches narrow that gap, harden the public payer page, and stop wasted API spend on visitors who never showed intent to pay:

**v1.4.13 — server-side and basic page-side tightenings:**

1. **First cron-side mempool poll: publish + 60s → publish + 15s** (`src/lib/invoices/payment-schedule.ts`, `src/app/(dashboard)/invoices/actions.ts`). The publish action now consumes `PRE_MEMPOOL_DELAYS_MS[0]` as a single source of truth.
2. **Public invoice page renders the txid + mempool.space link instantly on detection** — `PaymentWatcher.onStatusChange` now forwards the txid; `InvoicePaymentView` holds it in client state and renders without a manual refresh. Realtime payload also spreads `btc_txid` for the cron-only path.
3. *(v1.4.13 also lowered the WS-close fallback poll start from 10s to 2s. This fallback was removed entirely in v1.4.13.3, see below.)*

**v1.4.13.1 — visibility-aware active polling:**

4. **Active alongside-WebSocket REST poll** every 5s, gated on `paymentRevealed` (payer clicked "Pay now in Bitcoin" OR invoice already detected/paid) AND tab visibility. Caps at 60 polls (~5 min of visible time).

**v1.4.13.2 — close window-shopper API leak:**

5. **WS-close fallback gated on `paymentRevealed`.** Stopped exp-backoff polling for unrevealed visitors when the testnet WS died (~60s mark).

**v1.4.13.3 — remove vestigial fallback:**

6. **WS-close exp-backoff fallback removed entirely.** Real-world testing showed it overlapped with the v1.4.13.1 active poll, producing irregular polling clusters (1s/3s/4s gaps when both timers fired together). With the active poll covering the same cases, the fallback was vestigial. The single client-side polling path is now the active alongside-WS poll only.

**v1.4.13.4 — phased active-poll cadence:**

7. **Flat 5 s × 60 cadence replaced with a phased schedule** that mirrors the "Mark as Sent" button's front-loaded design but anchored on reveal. Total polls per real payment attempt drop from 60 to 25; the first minute (the dominant wallet-pay window) is unchanged at 5 s.

**v1.4.13.5 — tighter cron retry cadence:**

8. **Pre-mempool cron schedule tightened** from `[15 s, 5 min, 10 min, 30 min]` → `[15 s, 30 s, 60 s, 2 min, 5 min, 10 min, 30 min]`. The old schedule meant: when the cron's first poll at t=60s missed (mempool.space's indexer often hadn't yet surfaced the broadcast tx), the next attempt was 5 minutes out, leaving the dashboard stale for that whole window. New schedule does 4–5 polls in the first 5 minutes — a payer who closes the tab right after broadcasting now sees the invoice flip on the dashboard within ~2 min.

**v1.4.13.6 — save-time uniqueness check:**

9. **`saveDraft` and `updateDraft` now run the same address-uniqueness check that `loadAndAuthorise` (publish path) has run since v1.4.12.** Pre-v1.4.13.6, you could save a draft with a BTC address already used on another non-draft invoice; the conflict only surfaced at publish time. Now both paths reject up-front with the same error.

**v1.4.13.7 — skip redundant POSTs on already-detected invoices:**

10. **Watcher only POSTs to `/api/invoices/<id>/payment-status` on a real transition.** Pre-v1.4.13.7, every active-poll tick that found the same unconfirmed tx for an already-`payment_detected` invoice would re-POST `payment_detected` (~12 wasted POSTs per active payer over the 5-min polling window). Now the POST is skipped when local status is already at-or-past `payment_detected` and the watcher is just waiting for the confirmation transition. The GET to mempool.space still fires — that's needed to detect when the tx confirms.

What was *not* changed: Vercel Cron's 1/min minimum tick, the post-mempool retry cadence (10 min × 3, etc.), or the dashboard-side detection paths.

## Setup

```bash
set -a; source .env; set +a
```

You will need:
- The dev server running (`npm run dev`).
- A draft invoice you can publish, with a fresh testnet BTC address.
- Some testnet sats in a wallet you control (faucet or pre-funded).
- Browser DevTools open — Network tab + Console + Application > Service Workers.
- Confirm `NEXT_PUBLIC_BTC_NETWORK=testnet4` in `.env`.

For TESTS 1, 6, and 7 you will need to **broadcast a real testnet tx** to the invoice's BTC address. Plan to have your wallet ready before starting.

### ⚠️ Cron requirement for TESTS 7, 8, and 12 (cron-dependent tests)

**Vercel Cron does not fire locally.** In production, `vercel.json` schedules `/api/cron/payment-sweep` to run every minute, but in `npm run dev` nothing automatically calls it. To simulate this locally for tests that depend on the cron, run a polling loop in a separate terminal **before** starting the test:

```bash
# In a separate terminal — kill with Ctrl+C when done.
# Pulls CRON_SECRET from .env and hits the local sweep route every 60s.
set -a; source .env; set +a
while true; do
  echo "[manual-cron] $(date +%H:%M:%S) — calling sweep"
  curl -s -H "Authorization: Bearer $CRON_SECRET" http://localhost:3000/api/cron/payment-sweep
  echo
  sleep 60
done
```

You should see the cron route respond with a JSON summary like `{"processed":N,"transitions":0,"errors":0,"overdueFlips":0}` every minute. **If you don't see this output during your test, the cron-dependent tests will all fail because no server-side detection ever runs.**

For the v1.4.13.5 schedule-tightening tests specifically (TEST 7, TEST 8), the 60s loop interval matches production cadence and is what you want.

## ✏️ TEST 1 — Pay-and-don't-click: detection in ≤ 5s on the page (headline; updated in v1.4.13.4)

This is the v1.4.13.1+ win — the active alongside-WS poll closes the silent-WS gap. **Updated v1.4.13.4:** the cadence is now phased, not flat. Within the first minute it still polls every 5s, then progressively slows down.

**Steps:**
1. Publish a draft invoice with a fresh testnet BTC address.
2. Open the public invoice page (`/invoice/<id>`) in a fresh tab.
3. Click **"Pay now in Bitcoin"** to reveal the QR code and address.
4. Open DevTools → Network → filter `mempool.space` (broad — works on testnet and mainnet). **Do NOT** click "Mark as Payment Sent".
5. From your wallet, broadcast a tx to the invoice's address.
6. Start a stopwatch the moment your wallet says "broadcasting".

**Expected:**
- Within **~5 seconds** of broadcast (plus mempool.space's own indexing lag, typically 5–20s on testnet), the page flips to "Payment Detected".
- The Transaction ID and mempool.space link appear immediately, **without a page refresh**.
- In the Network tab during the first minute: `GET .../api/address/<addr>/txs` request every 5 s on a clean cadence. **No 1s/3s/4s clusters** — this was the v1.4.13.3 fix.
- After 60 s without a payment, the cadence drops to 10 s. After 120 s, 15 s. After 180 s, 30 s. After 240 s, 60 s. Total 25 polls over 5 min, then stops entirely.
- Total wall-clock detection (broadcast → status flip): typically 5–25s, well under the 75s reported pre-v1.4.13.1.

## ✏️ TEST 2 — Window-shopper: zero polling of any kind when address not revealed (updated for v1.4.13.2)

This proves API spend is bounded — viewers who never reveal don't burn quota, **even after the WebSocket dies** (which mempool.space's testnet WS does routinely after ~60s).

**Steps:**
1. Open a published invoice's public page in a fresh tab. **Do NOT** click "Pay now in Bitcoin".
2. Open DevTools → Network. **Important — use the right URL filter:** on testnet, the URL is `https://mempool.space/testnet4/api/address/<addr>/txs`. Filter on `mempool.space` (broad) or `api/address` (specific). Don't use `mempool.space/api/address` — the `/testnet4/` segment in between will cause the filter to miss everything.
3. Wait **at least 90 seconds** (long enough for the testnet WS to die and the v1.4.13 fallback to want to start).

**Expected:**
- **One** `GET https://mempool.space/testnet4/api/address/<addr>/txs` request immediately after page load — the watcher's mount-time check.
- A WebSocket connection to `wss://mempool.space/testnet4/api/v1/ws` (in the WS sub-tab).
- The WS may or may not stay alive — testnet routinely closes it after ~60s of inactivity.
- **After the WS dies:** ZERO further `txs` requests. (Pre-v1.4.13.2 you'd see exp-backoff polls at 2s, 4s, 8s, 16s, 32s, 64s after WS close. Those should now be absent.)
- Now click **"Pay now in Bitcoin"**. Within 5 seconds the active alongside-WS poll should start firing every 5s.

**If you DO see `txs` requests at 2/4/8/16/32/64s intervals after the WS dies, this is a regression** — please flag and don't ship.

## ✏️ TEST 7 — Cron-only path (tab closed entirely) — updated for v1.4.13.5

Tests the +15s first-check tightening AND the v1.4.13.5 tightened retry schedule. The retry tightening matters most when mempool.space's indexer doesn't surface the tx in time for the first cron poll — which happens routinely on testnet.

**⚠️ Required setup:** the manual-cron loop from the Setup section above must be running in a separate terminal **before** you start. Without it, this test will fail because the server-side cron never runs in dev.

**Steps:**
1. Confirm the manual-cron loop is running and you can see it print `[manual-cron] HH:MM:SS — calling sweep` followed by a JSON response every 60s.
2. Publish an invoice with a fresh address.
3. Open the public payer page in a new tab. Click "Pay now in Bitcoin" to reveal the address.
4. From your wallet, **broadcast a tx** to the invoice's address.
5. **Immediately close the public-page tab** (so no client-side detection paths are active).
6. Stay on the dashboard `/invoices` page (or open it in another tab).
7. Watch the invoice row's status badge.

**Expected (v1.4.13.5):**
- The invoice row's status badge should flip to **Payment Detected** within **~2 minutes** of broadcast in the typical case, even when mempool.space's indexer takes 60–120s to surface the tx.
- The flip happens because cron-side polls fire at minute boundaries: t+60s (often misses on testnet), t+120s (usually catches), t+180s, t+240s, t+300s, then 5 min, 10 min, 30 min.
- Pre-v1.4.13.5: the dashboard would have shown the stale `pending` status for **5 minutes** after the first cron miss because the next retry was 5 min out. That's the bug v1.4.13.5 fixes.

**Sanity check:**
- If you click into the invoice detail page mid-flow, the page's mount-time watcher also fires a fresh REST check (independent of cron). So if you're impatient, opening the detail page can short-circuit the wait.

**DB verification (optional):**
- Confirm in Supabase Studio: `SELECT id, status, stage_attempt, next_check_at, next_check_at - now() AS time_until_next_check, mempool_seen_at FROM invoices WHERE id = '<id>';`
- Right after publish: `next_check_at = published_at + 15s`, `stage_attempt = 0`.
- After first cron miss: `next_check_at = first_poll_time + 30s`, `stage_attempt = 1`.
- After second cron miss: `next_check_at = second_poll_time + 60s`, `stage_attempt = 2`.

## ✏️ TEST 8 — Pre-mempool schedule (v1.4.13.5)

Confirms the new schedule values *and* the `PRE_MEMPOOL_DELAYS_MS[0]` ↔ `publishStatePatch` consolidation.

**⚠️ Required setup:** the manual-cron loop from the Setup section above must be running in a separate terminal for the `next_check_at` and `stage_attempt` columns to evolve over time.

**Steps:**
1. Open `src/lib/invoices/payment-schedule.ts`. Confirm:
   - `PRE_MEMPOOL_DELAYS_MS[0] === 15_000` (publish-time first check)
   - `PRE_MEMPOOL_DELAYS_MS[1] === 30_000` (first miss → 30s)
   - `PRE_MEMPOOL_DELAYS_MS[2] === 60_000` (second miss → 60s)
   - `PRE_MEMPOOL_DELAYS_MS[3] === 120_000` (third miss → 2min)
   - `PRE_MEMPOOL_DELAYS_MS[4] === 300_000` (fourth miss → 5min)
   - `PRE_MEMPOOL_DELAYS_MS[5] === 600_000` (fifth miss → 10min)
   - `PRE_MEMPOOL_DELAYS_MS[6] === 1_800_000` (sixth miss → 30min, then stop)
2. Open `src/app/(dashboard)/invoices/actions.ts`. Confirm `publishStatePatch` reads `PRE_MEMPOOL_DELAYS_MS[0]` (no `60_000` or `15_000` literal).
3. Publish a draft invoice with no payment.
4. Run the SQL query below to inspect the row's polling state. **Three options, pick whichever you prefer:**

   **a) Supabase CLI (easiest from terminal):**
   ```bash
   # Linked cloud project. Drop --linked if you run Supabase locally.
   npx supabase db query --linked --output table \
     "SELECT id, status, stage_attempt, next_check_at, next_check_at - now() AS time_until_next_check, mempool_seen_at FROM invoices WHERE id = '<id>';"
   ```

   To watch it evolve every 30s:
   ```bash
   INV_ID="<id>"
   watch -n 30 "npx supabase db query --linked --output table \"SELECT id, published_at, next_check_at, stage_attempt FROM invoices WHERE id = '$INV_ID';\""
   ```

   **b) Supabase Dashboard SQL Editor (web UI):**
   Go to https://supabase.com/dashboard → your project → SQL Editor (sidebar). Paste and run:
   ```sql
   SELECT id, published_at, next_check_at, stage_attempt
   FROM invoices
   WHERE id = '<id>';
   ```

   **c) Direct psql (if you have it installed):**
   ```bash
   # Local Supabase: connection string from `npx supabase status`.
   psql "<DB_URL>" -c "SELECT id, status, stage_attempt, next_check_at, next_check_at - now() AS time_until_next_check, mempool_seen_at FROM invoices WHERE id = '<id>';"
   ```

   **The UUID must be wrapped in single quotes** in all three options. The shell prompt in your terminal does NOT speak SQL — running `SELECT ...` directly in zsh will give you `command not found: SELECT`.

**Expected (read the `time_until_next_check` column for the easiest visual confirmation):**
- Right after publish: `time_until_next_check ≈ 15 s`, `stage_attempt = 0`.
- After ~60 s (first cron pass with no tx): `time_until_next_check ≈ 30 s`, `stage_attempt = 1`.
- After ~120 s: `time_until_next_check ≈ 60 s`, `stage_attempt = 2`.
- After ~180 s: `time_until_next_check ≈ 2 min`, `stage_attempt = 3`.
- And so on through attempt 7, after which `next_check_at = NULL` and the row is hibernated until the user revisits or pays.

**Note:** `invoices` doesn't have an explicit `published_at` column. `created_at` is when the draft row was first created; the publish flow updates the draft row (status, `next_check_at`, `stage_attempt`, etc.) without writing a separate publish-time column. `time_until_next_check` is computed from `next_check_at - now()`, which is what you actually care about for verifying the schedule.

## ✏️ TEST 10 — Active poll engages automatically on already-detected/paid invoices (updated for v1.4.13.7)

Auto-revealed cases also count as "revealed" for polling purposes — but the v1.4.13.7 fix means already-detected invoices only fire **GET** requests, not POSTs.

**Steps:**
1. Manually flip a published invoice's status to `payment_detected` in the DB (or use a real flow). The address must have an unconfirmed paying tx in mempool.space.
2. Open the public page in a new tab.
3. Open DevTools → Network. Filter `mempool.space` (broad — catches the GET to mempool's API). Also keep an eye on requests to `/api/invoices/<id>/payment-status` — those are the POSTs to our own backend.

**Expected (v1.4.13.7):**
- The page auto-reveals the QR + address (no "Pay now in Bitcoin" button).
- **GET** requests to `mempool.space/.../api/address/<addr>/txs` fire on the v1.4.13.4 phased cadence (5 s × 12, 10 s × 6, 15 s × 4, 30 s × 2, 60 s × 1).
- **No POSTs** to `/api/invoices/<id>/payment-status` fire — local status is already `payment_detected`, so there's nothing meaningful to report.

## ✏️ TEST 12 — Window-shopper + WS death = total polling silence (v1.4.13.2 + v1.4.13.3)

This is the joint v1.4.13.2 + v1.4.13.3 reason-for-being. Pre-v1.4.13.2, window-shoppers would generate exp-backoff polling forever after the testnet WS died. v1.4.13.3 removed the fallback entirely, so the silence is structural — the code path that would have polled simply doesn't exist anymore.

**Note:** the manual-cron loop is *not* required for this test (it specifically asserts that *no* polling occurs from the page itself). But running it doesn't hurt.

**Steps:**
1. Open a published invoice's public page in a fresh tab. **Do NOT** click "Pay now in Bitcoin".
2. Open DevTools → Network → filter `mempool.space`. Confirm the mount-time `txs` request fires once.
3. Wait for mempool.space's testnet WebSocket to die naturally (~60–120 seconds — you'll see the WS row go from "pending" to "finished" or similar).
4. Continue waiting for **at least 5 more minutes** without revealing the address.

**Expected:**
- Approximately **one** `GET .../api/address/<addr>/txs` request total in a production build (the mount-time check from step 2). In `npm run dev` you may see 2–3 over a long session — likely caused by Fast Refresh / HMR re-mounting the watcher. This is dev-mode noise; see the "Known caveat" box below.
- The page still shows status = `pending`. If a payment arrives during this window via cron, the Supabase realtime subscription updates the badge.

**If you see GETs at a regular cadence (every 60 s or denser), this is a real regression and worth flagging.**

> **Known caveat (v1.4.13.7+, tracked in roadmap v1.4.21):** Manual testing of v1.4.13.7 showed 2 GETs spaced ~60–90 s apart in dev mode, even for window-shoppers. The trigger appears to be HMR re-mounting the `PaymentWatcher` component. To verify production behaviour, run `npm run build && npm run start` and re-test — the second GET should be absent. v1.4.21 is the planned investigation/fix; until then, treat occasional extra mount-time GETs in dev as benign.

**Now reveal:** click "Pay now in Bitcoin". Within 5 seconds, active polling should begin on the v1.4.13.4 phased cadence (5 s × 12, then 10 s × 6, etc., capped at 25 polls / ~5 min).

## 🆕 TEST 14 — Save-draft rejects an address already used on another non-draft invoice (v1.4.13.6)

Pre-v1.4.13.6, drafts could be saved with a BTC address already used on another active (non-draft) invoice in the user's account. The conflict only surfaced at publish time, producing a confusing fail-late UX. v1.4.13.6 makes save and publish enforce the same rule.

**Steps:**
1. Publish an invoice with a fresh testnet BTC address (call it `bc1qfresh1...`). Leave it in `pending` status — don't mark it paid or delete it.
2. Open `/invoices/new`. Tick "Accept Bitcoin" and paste in **the same address** `bc1qfresh1...`.
3. Click **Save**.

**Expected:**
- Save fails immediately with: **"This bitcoin address has already been used on invoice <ref>. Please provide a unique address."**
- The new draft is not created — the dashboard list shouldn't show a new row.

**Sanity checks:**
- Try the same flow with a freshly-generated address that's not used anywhere → save succeeds.
- Edit an existing draft (one that already uses an address) and update it without changing the address → save succeeds (the uniqueness query excludes the row being edited).

**Companion test (already in `manual-tests/v1.4.12-btc-address-hardening.md`):** TEST 9 in that doc asserts save also rejects an address with **on-chain history** (different rule, same fail-fast principle). v1.4.13.6 closes the symmetry: both rules now fire at save AND publish.

## Rollback notes

If you need to roll this version back:
- No DB migrations were added — only application code changed.
- Reverting the merge commit fully restores prior behaviour.
- Pre-existing `next_check_at` values on already-published invoices are unaffected (the change only applies to *new* publishes from the moment of the rollout).
- `PaymentWatcher`'s new `paymentRevealed` prop has a default of `false`, so any caller that stops passing it falls back to v1.4.13's WS-only mode without breaking.
